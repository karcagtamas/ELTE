# CPP - 2023-05-04

## Concurrent programming

- Elemi cel jobb es hatekonyabb parhuzamosithato programokat irni

### Problems

- A megirt kod sok optimalizacion esik at, nem ugyan azt a kodot generalja a fordito gepi kodban (optimalizacio)
- Parhuzamos CPU altali feldolgozas
  - Elterhetnek a CPU-k a gepi kod sorrendjetol, mert hatekonyabban tudja vegre hajtani
  - Branch prediction
  - A CPU kepes lehet ra, de akar a gepi kodban is lehet ilyeneket definialni
  - Eredmeny kiirasa a cache-be
    - Elerhetoseg a tobbi thread szamara
- Olyan eszko rendszer kell, amivel szabalyozott modon lehet elkesziteni a kodot es kihasznalja az optimalizaciokat
- Parhuzamos thread-ek nem feltetlen determinisztikus eredmeny nyujtottak
  - Neha az optimalizacio bezavarja a multi-thread kornyezetet
- Multi thread kornyezetbol, neha regi/uj memoria teruletet olvasnak a thread-ek a CPU cache-bol

### Singleton pattern

- Garantalja, hogy az adott tipusbol csak egy letezik
  - Static tarolas
  - ```singleton::instance()->method()``` fv
- Nem tokeletes
  - Az elso letrehozas a thread-ek egy ideju hivasa, akar ket hivast is indit + memory leak
- Locking-ra van szukseg
  - ```static Mutex lock_```
  - Lockolja egy szalon, a tobbi varakozik
- A Guard lock ne az instance() fv elejen legyen, hanem csak akkor, amikor letre is kell hozni
  - A lock blokkolhajta az olvasast is feleslegesen

#### Double cheked locking pattern

- Ell -> Lock (ha letre kell hozni) -> Ell -> Create instance
- Problema
  - Az instance inicializacio nem biztos hogy atomi, igy lehet hogy nem egy lepesban hajtodik vegre
  - Vagy a ket lepes kozott a pointer feleg keszult csak el, de ez mar eleg ahhoz hogy ne legyen nullptr, de meg invalid a ptr
- ```new``` expression
  - Bonyolult lepes
  - Memoria allokacio, konstruktor futtatas, majd pointer visszaadas
    - Optimalizacio celjabol ezeknek a lepeseknek a felcserelese

### Observable behaviour

- A kulso megfigyeles eredmenye ekvivalensnek kell lennie a meghatarozott programmal
- A kulvilag szamara lathato
  - I/O operaciok
  - Volatile objektumok irasa/olvasas
    - volatile objektumok nem lehetnek optimalizalva, sorrend nem modosithato

### Sequence point

- Inicializacio szetbontasa lepesekre
- Optimalizacio kiuti a szetbontast
  - volatile -ek haszanalat viszont segithet
  - ket volatile nem cserelhet fel, sorrendjuk nem valtoztathato meg
  - ket nem volatile, illetve egy volatile es egy nem volatile egymashoz viszonyitott sorrendje megvaltoztathato

### Modern hardware architecture

- Nem garantalja semmit, hogy a memoriaban mar inicializalva lett az objektum, vagy nem a regi adatokat tartalmazza, amik meg nem lettek kitorolve, de mar a torlesuk elindult

### Memory barrier

- Az utasitasok nem kerulhetnek a memoria tuzfalakon kivulre
  - Vannak felig atereszto megoldasok
    - Csak egyik iranyba enged optimalizaciot
  - A kulonbozo thread-eken sem
- ```Acquire()``` es ```Releases()```

### C++11 memoria modell

- Leirja a threadek kozotti interkaciokat a memorian keresztul
- Leirja a viselkedeseket
- Megfelelosegek a fordito szamara a kod generalasahoz
- Szerzodes a programozo es a rendszer kozott
- A fejleszto vallalja, hogy data race mentes programot ir, es a fordito garantalja a megfelelo kodot

### Terminologia

- Minimalis garanciak
  - A mar nem blokkolt thread tovabb halad
  - az implementacionak garantalnia, hogy az irasok egy threaden belul lathatonak kell lennie a tobb thread szamara "megfelelo idon belul"
- A operacio B elott van ha
  - egy threadban vannak es egymas utan futnak le (van leirva)
  - ket threadben ha a ket operacio kozott van egy szinkronizacios pont
- szinkronizacios pont
  - thread letrehozas szinkronizal egy masik thread vegrehajtasanak kezdetevel
  - thread vegreahajtas szinkronizal a join visszateresevel
  - unlock a mutex szinkronizal a kovetkezo lock-kal
- memoria hely
  - objektumok skalarisak
  - maximum szomszedos bitek sorozata nem null szeles
- Data race == undefined behaviour
  - Ket thread-ben ugyan azt a memoria helyet szeretnem elerni, legalabb az egyik nem atomikus es nincs A happens before kapcsolat
- Ket thread vegrehajtasa frissithet es hozzaferhet kulonbozo memoria helyekhez anelkul, hogy zavarnak egymast

### Szekvencialis konzisztencia

- Minden thread szekvencialisan van vegrehajtva
- Nem tapasztalhatja egy thread eredmenyeit valtozo sorrendben egy masik thread
- Szekvencialisnak tunik a tobbi thread szamara a begrehajtasa
- Ket fv-ben torteno lock es unlock
  - barmelyik lefuthat elobb
  - ket iranyi szinkronizacios pont => hibas mukodes is lehet
- ```std::atomic<bool> flag = false```
  - ```store()``` lock, unlock
  - ```load()``` lock, unlock
  - Figyel a memoria konzisztencia
  - Figyel, hogy ne lehessen az elotte levo muveleteket utana rakni
  - Figyel, hogy ne lehessen a mogotte levo muveleteket ele rakni
  - Egy lepesben tortenik
  - Szinkronizacios pont a store es a load kozott
  - A fordito berakja a memory barrier-eket ha szukseges

### Memory ordering

- Relaxed memory ordering
  - Semmilyen korlatozas
  - Csinalhatja ugy hogy nincsenek memory barrierek
  - Nem erdekelnek a sorrendek
  - Lehet hogy a kulonbozo thread-ek abszolut kolun latjak
  - Nagyon gyors
  - A sok ilyen lock lassithatja a vegrehajtast es eroforras igenyes (memoria iras, torles, olvasas)
- Acquire - release memory order
  - Nem olyan szigoru mint a sequentiel consistency
  - store-release operacio szinkronizalja, load-acquire operacio olvassa a tarolt adatot
  - store-release happens before load-acquire
  - Some platform: sokkal hatekonyabb kodot general
- Consume - release memory order
  - store-release operacio szinkronizal
  - load-consume csak azt frissiti a cachben, ami fugg a consume-olt elemtol
  - itt csak azok a valtozok kozott van A happens before B relacio, amik fuggenek a load-consume elemektol
  - mas olvasok eseten megjeleni a Data Race
  - Nem ajanlott a hasznalata

### std::thread

- Default konstruktor
  - Letrejon az objektum, de nem allokal az OP-tol
  - Es nem indul a thread
- Move konstruktor
  - Letezo thread atadja a jogait egy fizikai thread futatasahoz
- Nincs copy konstruktor
- Nincs assignment operator
- Template konstruktor
  - Lamda fuggvenyyel es argumentumokkal
- Beburkolja a std::thread a fizikai thread-et
- Lehet swapolni
- Az id es handle elrkerheto
- ```joinable()``` igaz ha meg nem lehet hivva join v. deatch
- ```join()``` ha a hivott thread befejezodott, akkor visszater, ha meg nem akkor bevarja a masik thread-et
- ```detach()``` lemond a gyerek thread-rol
- ha nem hivok sem join-t, sem detach-et es elkezd a fo szal destruktroalni, akkor exception-t dob (ha meg nem fejezodott be a gyerek thread - meg joinable)
- a thread argumentek mindig masolodnak (nem lehet ezt megakadalyozni, meg referenciaval sem) - atmasolodnak a thread local-ba
  - ```std::ref``` hasznalhato ennek a megoldasara
- Nem lehetnek ezek implicitek alapertelmezetten dekonstrualasnak
  - implicit join: nehez detektalni a performancia gondokat, destruktor var a thread-re
  - implicit detach: talan eltunnek az eroforrasok a gyerek thread alol
- Mukodnek kontenerekkel
- mem_fn
  - member pointert ment el es azt hivja meg operator(), amit elmentet

### Dead lock

- ```std::try_lock```

### Condition variable

- Ne nekem kelljen figyelnem, hanem szoljanak ha fel kell kellnem (thread)
- A producer oldal notify() hiv a consumer thread-ek szamara es azok fel kellnek
  - notify_one, notify_all
- cons_lock a mutexen (uniqe_lock)
- cond_wait megkapja a cons_lock-ot
  - megnezi a predikatum fv-t
    - thread safe kornyezet
  - ha hamis, akkor leveszi a lock-ot a cons_lock-rol es elalszik
  - ha igaz, akkor feldolgozza az adatot (vagy amit kell csinalni)
    - utana kell unlockolni
- nem a thread-nek kell figyelnie, hanem kap ertesitest ha fel kell kelnie
- spurious wake
  - felkelnek ertesites nelkul
  - nem meghatarozott ido es frekvenciankent
  - keruljuk a fuggvenyeket side effect-ekkel
